Elm Data Structures (3)

Elm,Frontend development,Data structures,Union Type

<p>Now that we have already looked at the data structures <a href="/articles/elm-data-structures-list-array-set-dict.html">lists, arrays, sets and dictionaries</a> as well as <a href="/articles/elm-data-structures-record-tuple.html">records and tuples</a>, there is only one more basic structure we need to talk about: The so called <em>Union Type</em>.</p> <div id="more"></div> <p>By defining a union type one always creates a new type that did not exist before. As opposed to the other data structures we already covered, a union type can be an amalgamation of different types – but it does not have to be.</p> <p>In its simplest form it can be thought of as an enumeration:</p> <pre><code class="language-elm"><span class="hljs-keyword">type</span> <span class="hljs-type">PullRequestState</span> = <span class="hljs-type">Proposed</span> | <span class="hljs-type">Rejected</span> | <span class="hljs-type">Merged</span> </code></pre> <p>This statement creates the type PullRequestState which can have one of the three mentioned values. These values are also called <em>tags</em> and are used to distinguish between the possibilities when handling the union type in a case expression. Depending on the tag, different actions can be performed:</p> <pre><code class="language-elm"><span class="hljs-title">branchColor</span> : <span class="hljs-type">PullRequestState</span> -&gt; <span class="hljs-type">String</span> <span class="hljs-title">branchColor</span> state = <span class="hljs-keyword">case</span> state <span class="hljs-keyword">of</span> <span class="hljs-type">Proposed</span> -&gt; <span class="hljs-string">"yellow"</span> <span class="hljs-type">Rejected</span> -&gt; <span class="hljs-string">"red"</span> <span class="hljs-type">Merged</span> -&gt; <span class="hljs-string">"green"</span> </code></pre> <p>So far, so unexciting – the fun begins when we start using compositions of different types. This way we can think of <a href="http://elm-lang.org/guide/model-the-problem">union types as state machines</a> or even various “classes” of entities.</p> <p>Let us examine this with an example in which we model the availability of a product:</p> <pre><code class="language-elm"><span class="hljs-keyword">module</span> Main <span class="hljs-keyword">exposing</span> (..) <span class="hljs-keyword">type</span> <span class="hljs-type">Availability</span> = <span class="hljs-type">SoldOut</span> | <span class="hljs-type">InStock</span> <span class="hljs-type">Int</span> | <span class="hljs-type">Reordered</span> ( <span class="hljs-type">Int</span>, <span class="hljs-type">Int</span> ) | <span class="hljs-type">Announced</span> <span class="hljs-type">String</span> <span class="hljs-title">displayStatus</span> : <span class="hljs-type">Availability</span> -&gt; <span class="hljs-type">String</span> <span class="hljs-title">displayStatus</span> availability = <span class="hljs-keyword">case</span> availability <span class="hljs-keyword">of</span> <span class="hljs-type">SoldOut</span> -&gt; <span class="hljs-string">"Sold out"</span> <span class="hljs-type">InStock</span> amount -&gt; <span class="hljs-string">"In stock: "</span> ++ (toString amount) ++ <span class="hljs-string">" left."</span> <span class="hljs-type">Reordered</span> days -&gt; <span class="hljs-keyword">let</span> min = toString (first days) max = toString (second days) <span class="hljs-keyword">in</span> <span class="hljs-string">"Available again in "</span> ++ min ++ <span class="hljs-string">" to "</span> ++ max ++ <span class="hljs-string">" days."</span> <span class="hljs-type">Announced</span> date -&gt; <span class="hljs-string">"Available on "</span> ++ date ++ <span class="hljs-string">"."</span> </code></pre> <p>This way we have the option of associating extra information to the tags. And these can even be different types of values, like in this example …</p> <ul> <li>an integer representing the amount of items left for the <code>InStock</code> tag</li> <li>a tuple of integers defining the range of days until the product will be available again for the <code>Reordered</code> tag</li> <li>a string representing the publishing date for the <code>Announced</code> tag</li> </ul> <pre><code class="language-elm"><span class="hljs-title">displayStatus</span> (<span class="hljs-type">InStock</span> <span class="hljs-number">42</span>) <span class="hljs-comment">-- "In stock: 42 left." : String</span> <span class="hljs-title">displayStatus</span> (<span class="hljs-type">Reordered</span> (<span class="hljs-number">3</span>,<span class="hljs-number">5</span>)) <span class="hljs-comment">-- "Available again in 3 to 5 days." : String</span> <span class="hljs-title">displayStatus</span> (<span class="hljs-type">Announced</span> <span class="hljs-string">"2016-12-24"</span>) <span class="hljs-comment">-- "Available on 2016-12-24." : String</span> </code></pre> <p>Each of these items might have a different shape, but as they share the same union type, we can combine them and have a unified way of handling them. For instance we can create a list of <code>Availability</code> items and display their states – regardless of their particular form:</p> <pre><code class="language-elm"><span class="hljs-title">availabilities</span> : <span class="hljs-type">List</span> <span class="hljs-type">Availability</span> <span class="hljs-title">availabilities</span> = [ <span class="hljs-type">SoldOut</span> , <span class="hljs-type">InStock</span> <span class="hljs-number">42</span> , <span class="hljs-type">Reordered</span> ( <span class="hljs-number">3</span>, <span class="hljs-number">5</span> ) ] <span class="hljs-type">List</span>.map displayStatus availabilities <span class="hljs-comment">-- ["Sold out","In stock: 42 left.","Available again in 3 to 5 days."] : List String</span> </code></pre> <p>This gives us an easy way of handling cases in which we would have used <a href="https://github.com/Dobiasd/articles/blob/master/from_oop_to_fp_-_inheritance_and_the_expression_problem.md">subclasses, when thinking in OOP terms</a>.</p> <p>That’s it: Now you know of the basic data structures in Elm. We will continue this series with looking at the type <code>Maybe</code> and examining pattern matching in more depth. After having covered these fundamentals we will dive into more advanced topics :)</p>
